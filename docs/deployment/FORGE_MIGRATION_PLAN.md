# KomunaID V2 — Laravel Forge Migration Plan

**Target audience:** Tech lead (Arie Andarean Atu) — concrete steps to move KomunaID V2 from Vercel to Laravel Forge (or equivalent VPS).

## 1. Why migrate

| Limitation on Vercel | On Forge / VPS |
|---|---|
| Cold start 600–900ms on first hit | Steady ~80–200ms |
| 60s function max duration | Unlimited (set PHP `max_execution_time`) |
| No long-lived queue worker | Native `php artisan queue:work` via Supervisor |
| `/tmp` ephemeral storage (logs/uploads vanish) | Persistent disk |
| Cron limited to 1/day, serverless crons only | OS cron, every minute |
| Free plan: 100 GB egress, throttling after burst | Flat monthly cost, no throttling |
| No SSH / no `php artisan tinker --execute` from local | Full SSH access |
| Serverless cost grows with traffic, not users | Flat cost ($12–25/mo) |

For KomunaID V2's needs (CMS, role dashboards, collaboration proposals, admin chat, premium demo, documentation), a small VPS is the right long-term home.

## 2. Target stack

- **Provider:** Laravel Forge + DigitalOcean (or Hetzner CX11 for cheapest)
- **Server size:** 2 vCPU / 2 GB RAM / 50 GB SSD (~$12-24/mo)
- **OS:** Ubuntu 24.04 LTS
- **PHP:** 8.2 (same as current, validates no upgrade surprises)
- **Web server:** Nginx 1.24
- **Database:** MySQL 8.0 (move from Hostinger `srv1761.hstgr.io` to a managed DO database or local MySQL)
- **Redis:** 7.x for cache + queue
- **Process manager:** Supervisor
- **TLS:** Let's Encrypt via Forge auto-renew
- **Backups:** Forge automatic daily snapshot (7-day retention) + DO managed DB backup

## 4. Step-by-step (Forge + DigitalOcean)

### 4.1 Provision

1. Sign up at https://forge.laravel.com ($12/mo for "Hobby" plan).
2. Connect DigitalOcean account.
3. Click **Create Server** → DO 2GB / Ubuntu 24.04 / Singapore (sin1) region.
4. Forge creates the droplet, runs the provisioning script (PHP 8.2, Nginx, MySQL, Redis, Supervisor, UFW), and sets up the `forge` sudo user.

### 4.2 Create site + database

1. In Forge: **New Site** → `komunaid.komuna.id` (or whatever domain).
2. PHP version: 8.2.
3. **New Database** named `komunaid` on the same server.
4. **New Database User** with full privileges on `komunaid`.

### 4.3 Deploy

1. Connect GitHub repo: `arieandreanatau/komunaid`.
2. Set deploy script:
   ```bash
   cd /home/forge/komunaid.komuna.id
   git pull origin main
   composer install --no-dev --optimize-autoloader
   php artisan migrate --force
   php artisan config:cache
   php artisan route:cache
   php artisan view:cache
   npm ci
   npm run build
   php artisan storage:link
   php artisan queue:restart
   ```
3. **Deployment Trigger:** "Deploy on push to main" (same as current Vercel).

### 4.4 Environment

Copy the Vercel env vars into Forge's **Environment** tab for the site. Map them to the new MySQL credentials:

```env
APP_NAME=KomunaID
APP_ENV=production
APP_KEY=base64:...   # same key
APP_DEBUG=false
APP_URL=https://komunaid.komuna.id

LOG_CHANNEL=stack
LOG_LEVEL=warning

DB_CONNECTION=mysql
DB_HOST=127.0.0.1
DB_PORT=3306
DB_DATABASE=komunaid
DB_USERNAME=forge
DB_PASSWORD=<generated by Forge>

CACHE_STORE=redis
SESSION_DRIVER=redis
SESSION_LIFETIME=120
QUEUE_CONNECTION=redis

REDIS_HOST=127.0.0.1
REDIS_PORT=6379
REDIS_PASSWORD=null

MAIL_MAILER=smtp
MAIL_HOST=smtp.postmarkapp.com
MAIL_PORT=587
...

CRON_SECRET=<openssl rand -base64 32>
```

### 4.5 Database import

```bash
# On local box
mysqldump -h srv1761.hstgr.io -u u519165229_arie -p u519165229_arie > komunaid-prod.sql
scp komunaid-prod.sql forge@<server-ip>:/home/forge/

# On Forge
mysql -u forge -p komunaid < /home/forge/komunaid-prod.sql
```

### 4.6 Queue worker

In Forge **Daemons** tab:

```
Command: php /home/forge/komunaid.komuna.id/current/artisan queue:work --sleep=3 --tries=3 --max-time=3600
User: forge
Processes: 1
```

This replaces the Vercel serverless function's implicit queue handling with a long-lived worker.

### 4.7 Scheduler

In Forge **Scheduler** tab, add:

```
* * * * * cd /home/forge/komunaid.komuna.id/current && php artisan schedule:run >> /dev/null 2>&1
```

This replaces the Vercel `crons` config in `vercel.json` that runs `/api/cron/scheduler?token=...` once a day. With OS cron you get per-minute resolution.

### 4.8 TLS + DNS

1. Forge auto-provisions Let's Encrypt cert.
2. In domain DNS panel, point A-record to Forge server IP.
3. Wait for propagation (1–60 min).

### 4.9 Smoke test

- Visit `https://komunaid.komuna.id/`
- Login as `superadmin@komuna.test / password`
- Verify dashboard, all modules, admin chat
- Place a small load test (Apache Bench or k6, 100 RPS for 1 min) — target P95 < 500ms
- Monitor Forge dashboard for CPU/RAM/disk

## 5. File / code changes required

### 5.1 `vercel.json`

Keep it for fallback; can be removed once cutover is verified.

### 5.2 `api/index.php` + `api/static.php`

The static-asset serving in `api/index.php` becomes dead code on Forge (Nginx serves `/assets/*` and `/build/*` directly). Can be left in place — they simply won't be hit — or removed for cleanliness.

### 5.3 `next.config.js` / `vite.config.js`

No changes needed.

### 5.4 `.env.production` template

Update with the new DB host, Redis host, and APP_URL. Document in `docs/deployment/FORGE_MIGRATION.md`.

## 6. Rollback plan

1. Keep Vercel deployment live until Forge has been verified for at least 48 hours.
2. DNS TTL: lower to 300s 24h before cutover so rollback is fast.
3. If Forge fails, point DNS back to Vercel — Vercel instance is still serving the same code and DB can be re-pointed.

## 7. Cost comparison

| Item | Vercel free | Vercel Pro | Forge + DO 2GB |
|---|---|---|---|
| Hosting | $0 | $20/mo | $12/mo (DO) + $12/mo (Forge Hobby) = $24/mo |
| DB | Hostinger (existing) | Hostinger (existing) | Included in DO droplet (MySQL on same box) or DO managed ($15/mo for HA) |
| Egress | 100 GB/mo included | 1 TB/mo included | 1 TB/mo DO included |
| Queue worker | None (db queue, no worker) | None | Supervisor (free) |
| Cron | 1/day | 1/day | per-minute |
| Cold start | yes | yes | no |
| SSH | no | no | yes |

**Forge + DO 2GB is $24/mo all-in and gives you a real Linux box** with persistent processes, predictable latency, and full SSH. That's the recommended next step.
